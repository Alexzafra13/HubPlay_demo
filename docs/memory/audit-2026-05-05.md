## 2. 🔴 P0 — Bugs / huecos accionables YA

Cada uno es independiente. Agrupar en una sola PR "hardening" o atacarlos por separado.

### P0-1 · `/health` devuelve **200 con DB caída**

- **Archivo**: `internal/api/handlers/health.go:27`
- **Síntoma**: Always writes 200; el fallo solo aparece en el JSON body (`"database": "error: …"`). Verificado por `health_test.go:52-71` (asserts 200 con DB error).
- **Por qué duele**: Kubernetes/HAProxy/uptime probes miran HTTP status code. Una BD muerta hoy es invisible para cualquier liveness/readiness check estándar. El día que despliegues tras un load balancer con healthcheck por status, esto te enmascara una caída total del servicio.
- **Fix**: Split en `/health/live` (proceso up) y `/health/ready` (deps — DB ping; opcionalmente ffmpeg, providers críticos). `/health/ready` retorna 503 si `db.Ping()` falla. Retain backwards-compat alias `/health` apuntando a `/live` o `/ready` según convenga.
- **Tests existentes a ajustar**: `health_test.go:52` pasa de "200 + error en body" a "503 + body explica qué falló".
- **Esfuerzo**: S, ~45 min.

### P0-2 · `RefreshToken` sin rate limit

- **Archivo**: `internal/auth/service.go:219` (`RefreshToken`)
- **Síntoma**: Solo `Login` (`internal/auth/service.go:167`) pasa por `loginRateLimiter`. `RefreshToken` no tiene gate.
- **Por qué duele**: Un refresh token leaked o guesseable se puede bruteforce indefinidamente. Combinado con que el flow de refresh tampoco tiene tests directos en `auth_test.go`, es un hueco doble.
- **Fix**: Reusar `loginRateLimiter` con sufijo distinto, o crear segundo `ratelimit.Limiter` keyed en `"refresh:" + ip + ":" + tokenPrefix`. Lockout 30 min tras 10 intentos fallidos (mismo perfil que Login). Guardar test en `auth_service_test.go` que asegure 429 tras N intentos.
- **Esfuerzo**: S, ~1h.

### P0-3 · `auth.Setup` (endpoint público "no users yet") sin test del gate

- **Archivo**: `internal/api/handlers/auth.go:302` (Setup handler)
- **Síntoma**: Una sola línea (`if count > 0 { return 403 }`) bloquea privilege escalation a admin para cualquier visitante no autenticado. **No hay test directo que asegure ese comportamiento**. Solo el wizard frontend lo prueba en e2e.
- **Por qué duele**: Si alguien flipea ese check a `>= 0` en un refactor (por ejemplo, refactorizando a interface o cambiando la lógica del wizard), nadie lo detecta. Es la única línea entre "deployment fresh" y "cualquier visitante registra admin".
- **Fix**: `auth_test.go` con dos casos: (a) Setup OK con count=0, (b) Setup 403 con count=1. Mock del repo trivial.
- **Esfuerzo**: S, ~30 min.

### P0-4 · Retention runners dangling

- **Archivos**:
  - `internal/iptv/service_epg.go:354` `CleanupOldPrograms` (cutoff 24h documentado, 0 callers).
  - `internal/db/federation_repository.go:669` `PruneAuditBefore` (comment literal: *"for completeness"*, 0 callers).
- **Síntoma**: `epg_programs` y `federation_audit_log` crecen monotónicamente. Cada M3U/EPG refresh appendea filas; cada peer request inserta audit log row.
- **Por qué duele**: En instalaciones long-lived estas tablas se convierten en el mayor surface de crecimiento de la DB. Un usuario con federation paireada + IPTV activa puede ver SQLite hinchado a múltiples GB en semanas.
- **Fix**: Daily ticker en un scheduler centralizado (mismo pattern que `auth.Service.StartSessionCleaner`). Wirearlo desde `cmd/hubplay/main.go` o desde los managers respectivos (`iptv.Scheduler`, `federation.Manager.bgCtx`). Cap configurable por YAML (e.g. `retention.epg_programs: 7d`, `retention.federation_audit_log: 30d`).
- **Esfuerzo**: S, ~1.5h (los dos juntos en un PR).

### P0-5 · `useUserDataSync` abre **3 EventSource al mismo `/me/events`**

- **Archivo**: `web/src/hooks/useUserDataSync.ts:85-87`. Cada llamada a `useUserEventStream` abre su propia `new EventSource(...)` en `useUserEventStream.ts:52`.
- **Síntoma**: Tres tipos de evento → tres conexiones SSE abiertas al mismo URL para cualquier usuario logueado. Los propios comments del código (`useEventStream.ts:23-25`, `useUserEventStream.ts:28-31`) ya documentan "promote to singleton with refcounts" — ahora es necesario, no opcional.
- **Por qué duele**: Chrome cuenta SSE contra el límite de **6/origin**. Sumando admin streams (LiveTV reload, scan progress) un admin puede tocar el cap y ver streams silenciados sin pista de error.
- **Fix**: `EventBus` singleton que mantenga 1 EventSource por URL con refcount sobre listeners `{type, handler}`. `useEventStream` y `useUserEventStream` quedan en wrappers thin.
- **Esfuerzo**: M, ~2h.

## 9. Notas finales

**El proyecto está en mejor estado del que parece.** Los fundamentos son sólidos: AppError envelope centralizado, observability federation con 5 collectors Prometheus, JWT keystore con rotación, sqlc bien adaptado (federation_repository migrado, ADR-001 cerrado), HLS lifecycle correcto en VOD, security headers, rate-limit en login, gosec gate en CI, Trivy scan, Dependabot. La cobertura de tests subió de ~55% a ~63% en backend y de 15% a ~27% en frontend. **Cero regresiones** desde el audit 2026-04-28.

Lo que queda abierto NO es deuda arquitectónica grave: son **god files que deben partirse**, **DTOs que deben tipificarse**, **5 huecos de hardening pragmático que se cierran en una sesión**, y **cobertura de pages frontend con acciones irreversibles** (admin pages). Ninguno bloquea el roadmap; todos suman calidad antes de etiquetar "production".

**Riesgo principal no mitigado**: ausencia de tests para 5 pages god/críticas (LiveTV, FederationAdmin, Settings, LibraryNewPage, SystemStatus). Cualquier refactor agresivo va a llevarse algo por delante sin red. Por eso las sesiones 2 y 3 incluyen tests del área que se toca.
