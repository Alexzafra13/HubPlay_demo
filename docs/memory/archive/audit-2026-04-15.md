## 1. Resumen ejecutivo

**HubPlay es un MVP funcional con buen esqueleto, superficie de docs inflada y 3–4 deudas concretas que conviene cerrar antes de seguir añadiendo features.**

- Backend (**118 .go**, **43 _test.go** ≈ 36%): wiring limpio en `main.go`, ~60 endpoints reales, dominio de errores consistente tras `AppError`, observabilidad Prometheus ya montada, keystore JWT con rotación ya implementado. Huecos: CSRF/Auth rompen el envelope, hay `panic(` en ruta de producción, `sqlc` está apuntado pero **no se usa** (repos escritos a mano), y el scanner/streaming tienen algunos hardcodes y leaks menores.
- Frontend (**~11.3k LOC**, **12 tests** ≈ 15%): casi todo real, no hay stubs. Algunos componentes grandes (`LiveTV.tsx` 828 líneas) y `Movies`/`Series` duplicados. Player HLS completo excepto selector de calidad.
- Docs (`docs/architecture/`, **11 688 líneas**, 25 ficheros): ~60% describen código que existe. `plugins.md` y `federation.md` son 100% aspiracionales — no hay una línea de código. `setup-wizard.md` y `overview.md` prometen endpoints y módulos que no existen.
- `CLAUDE.md` tiene al menos **3 datos falsos** (contados, no estimados): cifras de fichero, uso de sqlc, y directorios `plugin`/`federation` implicitos que no existen.

**Criterio de "proyecto limpio" (tu pedido)**:
1. Docs no prometen lo que el código no tiene.
2. `CLAUDE.md` refleja realidad verificable, no intención.
3. Cuatro bugs/deudas concretas cerradas.
4. Decisiones arquitectónicas tomadas quedan registradas como ADR.

El plan al final ordena las cosas por ese criterio.

## 2. Hallazgos críticos verificados (con referencias)

Estos son afirmaciones de **"esto no es lo que parece"**, no opiniones.

### 2.1 `sqlc` está listado pero no se usa

- `CLAUDE.md:14` dice: `Queries: sqlc (generated)`.
- `sqlc.yaml` existe en raíz y apunta a `internal/db/queries/`.
- **`internal/db/queries/` no existe.**
- `grep "Code generated by sqlc"` en `internal/db/` → **0 matches**.
- Los 16 repos son `database/sql` escritos a mano (`item_repository.go`, etc).
- `Makefile` tiene un target `sqlc:` pero es dead config.

**Implicación**: hay que decidir — o se activa sqlc de verdad y se migra, o se borra `sqlc.yaml` + target del Makefile + línea de CLAUDE.md y se documenta la decisión "repos a mano por X" en un ADR. La ambigüedad actual confunde.

### 2.2 Middleware CSRF y Auth rompen el envelope JSON

Cuatro sitios emiten errores con `http.Error()` y string JSON crudo, saltándose `respondAppError()` y el `ErrorRecorder` de observabilidad:

- `internal/api/csrf.go:61` → `CSRF_FAILED`
- `internal/auth/middleware.go:18` → `UNAUTHORIZED`
- `internal/auth/middleware.go:24` → `TOKEN_INVALID`
- `internal/auth/middleware.go:38` → `FORBIDDEN`

Consecuencia: estos errores no llegan a `hubplay_errors_total` en `/metrics`, no llevan `request_id` correlation, y el `Content-Type` queda `text/plain; charset=utf-8` en vez de `application/json` (defecto de `http.Error`). Todos los demás errores del API sí entran por el envelope.

### 2.3 `panic(` en ruta de producción

- `internal/api/csrf.go:81` → `panic("csrf: failed to generate random token: ..." )` si `crypto/rand.Read` falla.

En la práctica `crypto/rand` nunca falla en sistemas sanos, pero viola la convención "no panics en producción" y tumba el servidor en lugar de devolver 500. Trivial de cambiar por error + `respondAppError`.

### 2.4 Prometheus `// indirect` incorrecto

- `go.mod:25` → `github.com/prometheus/client_golang v1.23.2 // indirect`.
- Se importa directamente en 4 ficheros de `internal/observability/`.
- `go mod tidy` lo arregla. Ya estaba listado como pendiente en `project-status.md` — sigue sin hacerse.

### 2.5 `CLAUDE.md` con datos falsos/obsoletos

| Línea | Dice | Real |
|-------|------|------|
| 85 | 105 `.go` files | **118** |
| 85 | 38 `_test.go` | **43** |
| 14 | sqlc (generated) | **hand-written** (ver 2.1) |
| 96 | rama activa `claude/project-review-setup-QYnZs` | ya mergeada dos veces. Línea efímera |

El patrón es claro: los datos duros y efímeros en `CLAUDE.md` se oxidan cada sprint. Proponer: quitarlos y dejar solo convenciones estables.

## 3. Backend por paquete (resumen)

Detalle completo en los hallazgos de la auditoría; aquí solo lo accionable.

### `internal/api/` — ✔ con deuda puntual
- **Handlers consistentes** tras `respondAppError`/`handleServiceError`. Router bien estructurado, middleware en orden correcto.
- `progress.go:215,282,331` tiene campos `// backwards compat` **sin contexto**. Proyecto no está lanzado → borrables.
- `stream.go:222` construye session key con concatenación a mano; debería ser método del `stream.Manager`.
- Respuestas usan `map[string]any` en vez de DTOs tipados. Idiomático en Go pequeño, pero el frontend ya tiene los tipos en `web/src/api/types.ts`. Alinear ahorraría bugs.

### `internal/auth/` — ✔ sólido, un detalle
- JWT + keystore + rotación + bcrypt + rate-limit: todo presente y con tests.
- **No hay QuickConnect** pese a lo que diga `users-auth.md` (0 ocurrencias en código).
- `service.go:94–110`: cleanup de sesiones cada hora no registra error en DB failure.

### `internal/db/` — ✔ hand-written coherente
- 16 repos consistentes, helper `nullStr()` duplicado en varios (oportunidad de extraer).
- Sin batch operations: si algún día se escala, harán falta.
- **No hay retención** para `user_data` ni `epg_programs` (crecen sin límite).

### `internal/stream/` — ✔ waterfall real, algunos cabos sueltos
- `decision.go:31–92`: direct-play → direct-stream → transcode bien implementado.
- `manager.go:149`: sesiones **direct-play no se trackean** → `TouchSession`/`StopSession` fallan silentemente. Inconsistente.
- `transcode.go:42`: timeout hardcodeado a 4h; `config.TranscodeTimeout` existe pero `manager.go:70` lo ignora.
- `hls.go`: solo genera playlist, el serving de segmentos está en handlers (disperso).

### `internal/library/` + `internal/scanner/` — ✔ funciona, con puntos débiles
- Scheduler real, no stub. Scan async, mutex por library, context-aware.
- `scheduler.go:33–37`: contexto del scan de arranque mal acoplado al de Start() — si Start() cancela al volver, puede orfanar el scan inicial.
- `scanner.go:616`: fingerprint solo 64KB → **colisiones probables** con files grandes de cabecera similar.
- `scanner.go:332`: `enrichMetadata` best-effort → si TMDb cae durante un scan grande, items quedan sin metadata sin forma de marcarlos para reintento.
- `service.go:256–258`: `ScanAll` descarta errores sin propagarlos.

### `internal/iptv/` — ✔ todo real, 3 fugas
- M3U/XMLTV parsers + proxy con backoff. Funciona.
- `proxy.go:46–50`: el mapa de relays **no se purga** al desconectar. Memory leak bajo uso sostenido.
- `proxy.go:74–102`: reconnect retry infinito con cap 8s. Streams muertos consumen recursos para siempre.
- `service.go:54–65`: `RefreshM3U` hace `fetchURL()` dentro del Lock — bloquea otras ops mientras descarga.
- `xmltv.go:70`: parser XML sin protección XXE (riesgo bajo por contexto Docker, pero trivial de cerrar).

### `internal/provider/` — ✔ providers reales, **cero caching**
- TMDb, Fanart, OpenSubtitles: todos implementados con sus APIs correctas.
- **No hay caché** en ninguno. Cada scan de 1000 items = 1000+ requests a TMDb.
- No hay backoff ante `429`: el scanner silenciosamente descarta el item.
- `tmdb.go:89`: `Popularity/100` rompe rango 0–1 para blockbusters (≥100).
- `manager.go:88–91`: `Init()` sin timeout → un provider lento bloquea arranque.

### `internal/probe/`, `internal/imaging/`, `internal/blurhash/` — ✔ limpios
- `probe.go:198–209`: parse de framerate devuelve 0 silencioso ante error.
- `imaging/thumbnail.go`: nearest-neighbor funciona pero es lento para batch.
- `blurhash/`: implementación correcta; falta `Decode()` (no crítico, frontend no lo necesita).

### `internal/config/` — ✔ con una nota
- Preflight y validación completos. JWT auto-gen con fallback razonable.
- El comentario en CLAUDE.md de "auto-gen JWT secret" ya quedó obsoleto: **ahora el keystore de DB es la fuente de verdad**; `cfg.Auth.JWTSecret` solo se usa para bootstrap. Mencionar en ADR.

## 8. Reglas para no volver a este sitio

1. **Una doc en `docs/architecture/` = código verificable.** Si se empieza un diseño sin código, va a `docs/roadmap/`.
2. **`CLAUDE.md` solo con datos estables.** No cifras de LOC, no nombre de rama activa, no estado de sprint. Esos van a `docs/memory/project-status.md`.
3. **Antes de cerrar sesión de código**, correr:
   - `go mod tidy` si se tocaron imports.
   - `find internal -name '*.go' | wc -l` y actualizar `project-status.md` si cambió mucho.
   - `git grep -n 'TODO\|FIXME\|panic('` en el diff; si se introduce alguno, justificar.
4. **Cada decisión arquitectónica** (aunque sea pequeña) va como ADR corto a `architecture-decisions.md`. Incluso las "decisiones de no hacer" (ej: "sin sqlc por ahora").
5. **Tests primero en módulos nuevos**, mejor último en módulos existentes (es donde está el gap del 15% frontend).
